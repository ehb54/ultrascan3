585 /*
586  Inflate deflated (PKZIP's method 8 compressed) data. The compression
587  method searches for as much of the current string of bytes (up to a
588  length of 258) in the previous 32K bytes. If it doesn't find any
589  matches (of at least length 3), it codes the next byte. Otherwise, it
590  codes the length of the matched string and its distance backwards from
591  the current position. There is a single Huffman code that codes both
592  single bytes (called "literals") and match lengths. A second Huffman
593  code codes the distance information, which follows a length code. Each
594  length or distance code actually represents a base value and a number
595  of "extra" (sometimes zero) bits to get to add to the base value. At
596  the end of each deflated block is a special end-of-block (EOB) literal/
597  length code. The decoding process is basically: get a literal/length
598  code; if EOB then done; if a literal, emit the decoded byte; if a
599  length then get the distance and emit the referred-to bytes from the
600  sliding window of previously emitted data.
601 
602  There are (currently) three kinds of inflate blocks: stored, fixed, and
603  dynamic. The compressor deals with some chunk of data at a time, and
604  decides which method to use on a chunk-by-chunk basis. A chunk might
605  typically be 32K or 64K. If the chunk is uncompressible, then the
606  "stored" method is used. In this case, the bytes are simply stored as
607  is, eight bits per byte, with none of the above coding. The bytes are
608  preceded by a count, since there is no longer an EOB code.
609 
610  If the data is compressible, then either the fixed or dynamic methods
611  are used. In the dynamic method, the compressed data is preceded by
612  an encoding of the literal/length and distance Huffman codes that are
613  to be used to decode this block. The representation is itself Huffman
614  coded, and so is preceded by a description of that code. These code
615  descriptions take up a little space, and so for small blocks, there is
616  a predefined set of codes, called the fixed codes. The fixed method is
617  used if the block codes up smaller that way (usually for quite small
618  chunks), otherwise the dynamic method is used. In the latter case, the
619  codes are customized to the probabilities in the current block, and so
620  can code it much better than the pre-determined fixed codes.
621 
622  The Huffman codes themselves are decoded using a mutli-level table
623  lookup, in order to maximize the speed of decoding plus the speed of
624  building the decoding tables. See the comments below that precede the
625  lbits and dbits tuning parameters.
626  */

629 /*
630  Notes beyond the 1.93a appnote.txt:
631 
632  1. Distance pointers never point before the beginning of the output
633  stream.
634  2. Distance pointers can point back across blocks, up to 32k away.
635  3. There is an implied maximum of 7 bits for the bit length table and
636  15 bits for the actual data.
637  4. If only one code exists, then it is encoded using one bit. (Zero
638  would be more efficient, but perhaps a little confusing.) If two
639  codes exist, they are coded using one bit each (0 and 1).
640  5. There is no way of sending zero distance codes--a dummy must be
641  sent if there are none. (History: a pre 2.0 version of PKZIP would
642  store blocks with no distance codes, but this was discovered to be
643  too harsh a criterion.) Valid only for 1.93a. 2.04c does allow
644  zero distance codes, which is sent as one code of zero bits in
645  length.
646  6. There are up to 286 literal/length codes. Code 256 represents the
647  end-of-block. Note however that the static length tree defines
648  288 codes just to fill out the Huffman codes. Codes 286 and 287
649  cannot be used though, since there is no length base or extra bits
650  defined for them. Similarly, there are up to 30 distance codes.
651  However, static trees define 32 codes (all 5 bits) to fill out the
652  Huffman codes, but the last two had better not show up in the data.
653  7. Unzip can check dynamic Huffman blocks for complete code sets.
654  The exception is that a single code would not be complete (see #4).
655  8. The five bits following the block type is really the number of
656  literal codes sent minus 257.
657  9. Length codes 8,16,16 are interpreted as 13 length codes of 8 bits
658  (1+6+6). Therefore, to output three times the length, you output
659  three codes (1+1+1), whereas to output four times the same length,
660  you only need two codes (1+3). Hmm.
661  10. In the tree reconstruction algorithm, Code = Code + Increment
662  only if BitLength(i) is not zero. (Pretty obvious.)
663  11. Correction: 4 Bits: # of Bit Length codes - 4 (4 - 19)
664  12. Note: length code 284 can represent 227-258, but length code 285

666  since it gets used a lot in very redundant files. The length
667  258 is special since 258 - 3 (the min match length) is 255.

669  single stream of lengths. It is possible (and advantageous) for
670  a repeat code (16, 17, or 18) to go across the boundary between

769 /* Huffman code decoding is performed using a multi-level table lookup.
770  The fastest way to decode is to simply build a lookup table whose
771  size is determined by the longest code. However, the time it takes
772  to build this table can also be a factor if the data being decoded
773  is not very long. The most common codes are necessarily the
774  shortest codes, so those codes dominate the decoding time, and hence
775  the speed. The idea is you can have a shorter table that decodes the
776  shorter, more probable codes, and then point to subsidiary tables for
777  the longer codes. The time it costs to decode the longer codes is
778  then traded against the time it takes to make longer tables.
779 
780  This results of this trade are in the variables lbits and dbits
781  below. lbits is the number of bits the first level table for literal/
782  length codes can decode in one step, and dbits is the same thing for
783  the distance codes. Subsequent tables are also less than or equal to
784  those sizes. These values may be adjusted either when all of the
785  codes are shorter than that, in which case the longest code length in
786  bits is used, or when the shortest code is *longer* than the requested
787  table size, in which case the length of the shortest code in bits is
788  used.
789 
790  There are two different values for the two tables, since they code a
791  different number of possibilities each. The literal/length table
792  codes 286 possible values, or in a flat code, a little over eight
793  bits. The distance table codes 30 possible values, or a little less
794  than five bits, flat. The optimum values for speed end up being
795  about one bit more than those, so lbits is 8+1 and dbits is 5+1.
796  The optimum values may differ though from machine to machine, and
797  possibly even between compilers. Your mileage may vary.
798  */

825  tables to decode that set of codes. Return zero on success, one if
826  the given code set is incomplete (the tables are still built in this
827  case), two if the input is invalid (all zero length codes or an
828  oversubscribed set of lengths), and three if not enough memory. */

834  int h; /* table level */

836  register unsigned j; /* counter */
837  register int k; /* number of bits in current code */
838  int l; /* bits per table (returned in m) */

843  unsigned v[N_MAX]; /* values in order of bit length */

852  memzero( c, sizeof(c) );

854  i = n;

859  // n-i, *p));

861  p++; /* Can't combine with above line (Solaris bug) */

864  if ( c[0] == n ) /* null input--all zero length codes */

867  *m = 0;
868  return 0;

909  j = 0;

911  xp = x + 2;

920  i = 0;

939  z = 0; /* ditto */

968  }
969

972  }
973  }

988  u[h] = ++q; /* table starts after link */

998  u[h-1][j] = r; /* connect to last table */
999  }
1000  }
1001 
1002  /* set up table entry in r */
1003  r.b = (uch)(k - w);
1004 
1005  if (p >= v + n)
1006  {
1007  r.e = 99; /* out of values--invalid code */
1008  }
1009  else if (*p < s)
1010  {
1011  r.e = (uch)(*p < 256 ? 16 : 15); /* 256 is end-of-block code */
1012  r.v.n = (ush)(*p); /* simple code is just the value */
1013  p++; /* one compiler does not like *p++ */
1014  }
1015  else
1016  {
1017  r.e = (uch) e[*p - s]; /* non-simple--look up in lists */
1018  r.v.n = d[*p++ - s];
1019  }
1020

1023

1027  }

1030  for ( j = 1 << (k - 1); i & j; j >>= 1 )

1033  }
1034 
1035  i ^= j;
1036

1038  while ( (i & ((1 << w) - 1)) != x[h] )

1040  h--; /* don't need to update q */
1041  w -= l;
1042  }
1043  }
1044  }
1045

1047  return y != 0 && g != 1;

1706  * We use a lazy evaluation for matches: a match is finally adopted only if
1707  * there is no better match at the next window position. */
1708 
1709 #define HASH_BITS 15
1710  /* For portability to 16 bit machines, do not use values above 15. */
1711 
1712 #define MIN_LOOKAHEAD ( MAX_MATCH + MIN_MATCH + 1 )
1713 /* Minimum amount of lookahead, except at the end of the input file.
1714  * See deflate.c for comments about the MIN_MATCH + 1. */
1715

1717 /* In order to simplify the code, particularly on 16 bit machines, match
1718  * distances are limited to MAX_DIST instead of WSIZE. */

1721 #define HASH_MASK ( HASH_SIZE - 1 )

1724

1728 /* Number of bits by which ins_h and del_h must be shifted at each
1729  * input step. It must be such that after MIN_MATCH steps, the oldest
1730  * byte no longer takes part in the hash key, that is:
1731  * H_SHIFT * MIN_MATCH >= HASH_BITS */
1732 
1733

1735  * Update a hash value with the given input byte
1736  * IN assertion: all calls to to UPDATE_HASH are made with consecutive

1766 {

1768  IPos prev_match; /* previous match */

1770  int match_available = 0; /* set if previous match exists */
1771  register unsigned match_length = MIN_MATCH - 1; /* length of best match */
1772 
1773  /* Process the input block. */
1774  while ( lookahead != 0 )
1775  {
1776  /* Insert the string window[strstart .. strstart+2] in the
1777  * dictionary, and set hash_head to the head of the hash chain: */
1778 
1779  INSERT_STRING( strstart, hash_head );
1780 
1781  /* Find the longest match, discarding those <= prev_length. */
1782 
1783  prev_length = match_length, prev_match = match_start;
1784  match_length = MIN_MATCH-1;
1785 
1786  if ( hash_head != (ush) 0 &&
1787  prev_length < max_lazy_match &&
1788  strstart - hash_head <= MAX_DIST )
1789  {
1790  /* To simplify the code, we prevent matches with the string
1791  * of window index 0 (in particular we have to avoid a match
1792  * of the string with itself at the start of the input file). */
1793 
1794  match_length = longest_match( hash_head );
1795 
1796  /* longest_match() sets match_start */
1797 
1798  if ( match_length > lookahead ) match_length = lookahead;
1799 
1800  /* Ignore a length 3 match if it is too distant: */
1801 
1802  #define TOO_FAR 4096
1803  if ( match_length == MIN_MATCH && strstart - match_start > TOO_FAR )
1804  {
1805  /* If prev_match is also MIN_MATCH, match_start is garbage
1806  * but we will ignore the current match anyway. */
1807 
1808  match_length--;
1809  }
1810  }
1811 
1812  /* If there was a match at the previous step and the current
1813  * match is not better, output the previous match: */
1814 
1815  if ( prev_length >= MIN_MATCH && match_length <= prev_length )
1816  {
1817  flush = ct_tally( strstart - 1 - prev_match, prev_length - MIN_MATCH );
1818 
1819  /* Insert in hash table all strings up to the end of the match.
1820  * strstart-1 and strstart are already inserted. */
1821 
1822  lookahead -= prev_length - 1;
1823  prev_length -= 2;
1824 
1825  do
1826  {
1827  strstart++;
1828  INSERT_STRING(strstart, hash_head);
1829 
1830  /* strstart never exceeds WSIZE-MAX_MATCH, so there are
1831  * always MIN_MATCH bytes ahead. If lookahead < MIN_MATCH
1832  * these bytes are garbage, but it does not matter since the
1833  * next lookahead bytes will always be emitted as literals. */
1834  } while ( --prev_length != 0 );
1835 
1836  match_available = 0;
1837  match_length = MIN_MATCH - 1;
1838  strstart++;
1839 
1840  if ( flush ) FLUSH_BLOCK(0), block_start = strstart;
1841 
1842  }
1843  else if ( match_available )
1844  {
1845  /* If there was no match at the previous position, output a
1846  * single literal. If there was a match but the current match
1847  * is longer, truncate the previous match to a single literal. */
1848 
1849  if ( ct_tally ( 0, window[ strstart - 1 ] ) )
1850  {
1851  FLUSH_BLOCK(0), block_start = strstart;
1852  }
1853 
1854  strstart++;
1855  lookahead--;
1856  }
1857  else
1858  {
1859  /* There is no previous match to compare with, wait for
1860  * the next step to decide. */
1861 
1862  match_available = 1;
1863  strstart++;
1864  lookahead--;
1865 
1866  /* Make sure that we always have enough lookahead, except
1867  * at the end of the input file. We need MAX_MATCH bytes
1868  * for the next match, plus MIN_MATCH bytes to insert the
1869  * string following the next match. */
1870 
1871  } while ( lookahead < MIN_LOOKAHEAD && ! eofile ) fill_window();
1872  }
1873 
1874  if ( match_available ) ct_tally ( 0, window[strstart -1 ] );
1875 
1876  return FLUSH_BLOCK(1); /* eof */
1877 }
